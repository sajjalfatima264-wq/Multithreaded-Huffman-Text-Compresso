# Multithreaded Huffman Text Compressor

A C implementation of Huffman coding for text file compression, with multithreaded character counting and compression to compare performance across different thread counts.

## How it works

The pipeline runs in four stages:

1. **Histogram** (`sectioned_histogram.c`) — Splits the input file into roughly equal chunks (one per thread), counts character frequencies in parallel, then aggregates the results into `char_frequency.txt`. Also reports detailed timing for each stage (file read, thread creation, worker execution, aggregation, etc.) so you can benchmark scaling.
2. **Encoding table** (`huffman.c`) — Reads `char_frequency.txt`, builds a Huffman tree with a min-heap, and writes out `encoding.txt` — a sorted table of each character, its frequency, and its binary Huffman code.
3. **Compression** (`compress.c`) — Reads the original file and `encoding.txt`, splits the file into sections (one per thread), and has each thread independently encode its section into a bitstream. The sections are written sequentially to a binary output file, each prefixed with its bit count.
4. **Decompression** (`decompress1.c`) — Reads the compressed file section by section and walks each section's bitstream, matching prefixes against the code table to reconstruct the original characters, writing them back out to a text file.

## Files

| File | Purpose |
|---|---|
| `sectioned_histogram.c` | Multithreaded character frequency counter; outputs `char_frequency.txt` and a timing report |
| `huffman.c` | Builds the Huffman tree from frequencies; outputs `encoding.txt` |
| `compress.c` | Multithreaded compressor; encodes a text file into a binary file using the Huffman codes |
| `decompress1.c` | Decompressor; reconstructs the original text from a compressed file |
| `g2.py` | Plots completion time vs. number of threads using matplotlib |
| `commands.txt` | Shell one-liner to sort `char_frequency.txt` by frequency |
| `char_frequency.txt` | Generated character frequency table (semicolon-separated `char,count` pairs) |
| `encoding.txt` | Generated Huffman code table (char, frequency, binary code) |
| `text.txt` / `large.txt` | Sample input text |
| `compress.bin` | Compressed output |
| `decompress_text.txt` | Decompressed output (should match the original input) |
| `threads_graph.png` | Completion time vs. thread count, generated by `g2.py` |

## Building

Each program is standalone and compiles with `gcc`. The histogram, compression, and decompression programs link against pthreads:

```bash
gcc -O2 -o sectioned_histogram sectioned_histogram.c -lpthread
gcc -O2 -o huffman huffman.c
gcc -O2 -o compress compress.c -lpthread
gcc -O2 -o decompress1 decompress1.c -lpthread
```

## Usage

Run the four stages in order on your input file:

```bash
# 1. Count character frequencies (4 threads in this example)
./sectioned_histogram text.txt 4

# 2. Build the Huffman encoding table
./huffman char_frequency.txt

# 3. Compress the file (4 threads)
./compress text.txt encoding.txt 4 compress.bin

# 4. Decompress back to text
./decompress1 compress.bin encoding.txt 1 decompress_text.txt
```

Afterward, verify the round-trip was lossless:

```bash
diff text.txt decompress_text.txt
```

To inspect or re-sort the frequency table:

```bash
bash commands.txt   # or run the tr/sort/paste pipeline inside it directly
```

To regenerate the thread-scaling plot:

```bash
python3 g2.py
```

## Notes

- `sectioned_histogram.c` and `compress.c` parallelize across the requested number of threads; `decompress1.c` currently always decompresses one section at a time on a single thread, regardless of how compression was threaded — the section boundaries from compression are preserved in the compressed file format so this still works correctly, just not in parallel.
- Frequency and encoding files use `\n`, `\r`, and `' '` as special-cased labels for newline, carriage return, and space, since these can't be written as bare characters in the text tables.
- The thread count passed to `compress` does not need to match the thread count used for `sectioned_histogram` — they're independent stages.
